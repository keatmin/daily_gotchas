Usage of asyncio makes things easy when it works, but understanding what is happening and what is behind will help in designing better async app.

await and async#

await in an async function tells tells the event loop to suspend exec of g() and do something else


async def g():    r = await f()    return r

Hence using await is creating a coroutine function. Calling a coroutine function requires await

yield will give an async generator. Which can be used with async for

queue in asyncio#


async def produce(i: int, q: asyncio.Queue) -> None:    result = await f()    await q.put(result)
async def consume(i: int, q: asyncio.Queue) -> None:    tasks = await q.get()    try:         await do_task(tasks)    except exception as e:        print(e)        # return task to queue, use a counter to avoid running it infinitely        await q.put(i)    q.task_done()
async def main(n: int):    q = asyncio.Queue    # Fires up both producers and consumers    producers = [asyncio.create_task(produce(i, q)) for i in range(n)]    consumers = [asyncio.create_task(consume(i,q)) for i in range(n)]     await asyncio.gather(*producers) # Wait for producers to finish while both are running    await q.join() # This awaits for consumers implicitly and the rest of the tasks to be processed    for c in consumers:        c.cancel() # To cancel idle consumers when the queue is done

what about get_event_loop()?#

Older examples use asyncio.get_event_loop(), mainly a pre Python 3.7 version. With Python 3.7, asyncio.run(main()) replaces the need of having to use get_event_loop() which typically has a flow like:

loop = asyncio.get_event_loop(main())try:    loop.run_until_complete(main())finally:    loop.close()

Unless there's a need to go into a lower level to fine tune control in the event loop, usually an asyncio.run() is sufficient. The usage of loop.is_running() and loop.is_closed() is a good way to introspect loop and scheduling a callback.

Other functions#

asyncio.create_task()#

create_task() introduced in Python 3.7 is used to schedule the execution of coroutine object. Prior to that it was ensure_future()

async def do_task(a: list) -> list:    b = await task_c(a)    return b
async def main():    # For the sake of an example, this is done when in fact normally it can be done via asyncio.run(do_task(some_list)) or await do_task()    t = asyncio.create_task(do_task(some_list))    await t # Important to await t as asyncio.run() will run and will assume main is completed
asyncio.run(main())

asyncio.gather()#

This function just puts all the coroutines into one future returning only one future object. Essentially putting tasks into a list and waiting for it to be completed in the order it was created.

results = await asyncio.gather(return_exceptions=True) #To return exceptions in the list

monitoring with events#

Using event to do specific actions, if not done, extend time (if there's a timeout)

async def extend(msg, event):    while not event.is_set():        logging.info("Extend")        await asyncio.sleep(2)    else:        await cleanup(msg)

async def handle_message(msg):    event = asyncio.Event()    asyncio.create_task(extend(msg, event))    await asyncio.gather(do_task(msg), do_task_b(msg))    event.set()

good to know#

  1. Coroutines are useless unless in event loop
  2. The default of asyncio is running in single thread and single CPU core
  3. async context manager is defined in .__aenter()__ and .__a.exit()__
  4. If awaited task is CPU bound, running it in loop.run_in_executor() will allow it to run in its own process
  5. Putting everything in async is useless if all of the functions have blocking calls.
  6. Be sure to retrieve exceptions
  7. asychronous != concurrency, serial != blocking
  8. If using loop, cleanup using loop.close()
  9. Iterating is blocking
